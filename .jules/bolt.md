## 2024-05-27 - [Lazy initialization of Hook objects in TrieNode]
 **Learning:** In `core/trie.js`, each `TrieNode` initializes an object of 6 `Hook` instances (`onMatched`, `onNotMatched`, etc.) upon instantiation. This creates a large memory overhead and instantiation penalty for large tries or frequent insertions, where 100k insertions take ~370ms compared to ~38ms with lazy initialization. A memory guideline specifically warns: "To avoid significant overhead when creating frequent data structure nodes (like TrieNode in core/trie.js), avoid aggressively instantiating complex objects (like Hook instances) in the constructor. Lazily initialize them via a getter, and use an internal backing property (e.g., ._hooks?.) in performance-critical paths to prevent unintended object allocations."
 **Action:** Refactor `TrieNode` in `core/trie.js` to initialize the `hooks` object lazily via a getter property, storing it in a private or internal property `_hooks`. Modify `core/keyProcessor.js` to prefer checking `_hooks` (if present) before triggering them, or utilize optional chaining (`this.currentNode._hooks?.onMatched?.trigger(...)`) to prevent accidental creation of the `Hook` objects on nodes that haven't registered listeners.
